# 小铁 Agent 生产化迭代方案（2026）

## 1. 目标

围绕功能完整性、响应速度、稳定性、可扩展性、可维护性、用户体验、安全合规七个维度，构建可持续迭代的生产化版本，并建立可验证的工程基线。

## 2. 需求分析

### 2.1 功能完整性

- 支持多模型与工具编排
- 支持并行工具调用与可观测事件流
- 支持代理抓包与数据导出

### 2.2 性能与稳定性

- 降低高频事件发布的调度开销
- 降低高流量抓包时的队列淘汰成本
- 修复运行时配置切换与执行路径不一致问题

### 2.3 安全合规

- 对工具调用维持最小权限原则
- 对 MCP 接入执行显式授权与审计
- 对提示注入、敏感信息泄漏、过度代理执行建立防护

### 2.4 运维可观测

- 统一指标、日志、链路追踪语义
- 形成离线压测基线与回归门禁
- 提供一键部署与监控启动入口

## 3. 技术选型

### 3.1 编排与状态

- 当前代码以自研 Agent 循环为核心，短期保留该主干
- 中期可引入状态机式编排层作为可选模式，用于复杂分支与恢复

### 3.2 可观测性

- 采用 OpenTelemetry 语义作为指标与链路标准
- 部署侧使用 OTel Collector + Prometheus + Grafana

### 3.3 安全基线

- 安全风险建模以 OWASP LLM Top 10 2025 为基准
- MCP 接入遵循最新规范中的授权与同意要求

## 4. 模块重构与优化项

### 4.1 已落地（本次实现）

- 事件总线新增批量发布接口 `publish_batch`，减少高频 token 事件下的发布开销
- Agent 流式输出改为批量刷新模型，减少 flush 任务风暴
- 修复 `/parallel` 运行时切换仅改旧字段的问题，确保与执行分支一致
- RequestStorage 改为 `deque(maxlen)`，将满队列淘汰从 O(n) 降为 O(1)

### 4.2 下一阶段

- 增加工具调用级熔断与预算控制
- 增加长会话上下文分层与检查点恢复
- 增加统一策略引擎，对高风险工具调用执行二次确认

## 5. 算法与执行策略优化

- 事件层：从逐事件异步发布转为按阈值批量发布
- 存储层：用有界双端队列保证高负载稳定吞吐
- 调度层：并行开关采用单一事实源，避免配置漂移

## 6. 测试验证策略

### 6.1 单元测试

- 事件系统：验证批量发布完整性与顺序
- 命令系统：验证并行开关在 Agent 与 config 双路径一致

### 6.2 集成测试

- 复用现有 `test_proxy_integration.py` 与核心单元集
- 回归高风险文件：`test_sandbox.py`、`test_schema.py`、`test_orchestrator.py`

### 6.3 性能基准

- 提供独立基准脚本，输出批量事件发布与存储吞吐指标
- 形成基线文件，作为后续 PR 回归对比依据

## 7. 部署与运维

- 提供 Docker Compose 生产模板
- 提供 OTel Collector、Prometheus、Grafana 监控配置
- 提供统一部署脚本，执行构建、测试、监控栈拉起

## 8. 里程碑验收标准

- 功能完整性：核心链路测试通过
- 响应速度：批量事件与存储淘汰性能优于旧版
- 稳定性：全量测试通过，无新增高危失败
- 可扩展性：模块边界清晰，可继续接入策略层与状态层
- 可维护性：新增测试覆盖关键修复点
- 用户体验：流式输出平滑，无明显卡顿
- 安全合规：风险面有明确控制点与文档化策略
